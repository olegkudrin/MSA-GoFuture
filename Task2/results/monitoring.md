# Мониторинг Event-Driven архитектуры

## Обоснование подхода к мониторингу

Проблемы мониторинга EDA:

- Асинхронность: события обрабатываются асинхронно, сложнее отследить полный путь запроса
- Распределенность: события проходят через несколько сервисов и топиков Kafka
- Задержки: важно отслеживать задержки между публикацией и обработкой событий
- Порядок обработки: необходимо гарантировать порядок обработки событий в партициях
- Надежность: критично отслеживать потери событий, дубликаты, события в DLQ

Мониторинг должен обеспечивать:

- Наблюдаемость: понимание состояния системы в реальном времени
- Проактивность: обнаружение проблем до их влияния на пользователей
- Быстрое восстановление: быстрая диагностика и решение проблем

## Выбранные инструменты мониторинга

### Prometheus

Обоснование:

- Стандарт де-факто для метрик в микросервисных архитектурах
- Pull-модель сбора метрик (надежнее push при сбоях)
- Мощный язык запросов PromQL для анализа
- Интеграция с Alertmanager для алертинга
- Уже используется в текущей инфраструктуре

Использование:

- Сбор метрик от всех сервисов (latency, error rate, throughput)
- Сбор метрик от Kafka через Kafka Exporter
- Хранение временных рядов метрик
- Источник данных для Grafana дашбордов

### Grafana

Обоснование:

- Единый интерфейс для визуализации метрик, логов и трассировок
- Интеграция с Prometheus, Loki и Jaeger
- Гибкие дашборды с различными типами визуализации
- Интеграция с Alertmanager для уведомлений
- Корреляция данных из разных источников
- Уже используется в текущей инфраструктуре

Использование:

- Визуализация метрик из Prometheus
- Просмотр логов из Loki
- Анализ трассировок из Jaeger
- Создание кастомных дашбордов для мониторинга EDA
- Настройка алертов на основе метрик

### Loki

Обоснование:

- Централизованное хранение логов всех сервисов
- Интеграция с Grafana для единого интерфейса
- Эффективное хранение (компрессия, индексация)
- Уже используется в текущей инфраструктуре

Использование:

- Агрегация логов от всех сервисов
- Логи Kafka брокера, консьюмеров и продюсеров
- Поиск по логам через LogQL
- Корреляция логов с метриками в Grafana

### Jaeger

Обоснование:

- Распределенная трассировка критична для EDA
- Отслеживание пути событий через несколько сервисов
- Выявление узких мест в обработке событий
- Интеграция с OpenTelemetry (стандарт индустрии)

Использование:

- Трассировка событий от публикации до обработки
- Трассировка Saga оркестрации
- Анализ задержек между шагами обработки

### Schema Registry (Confluent Schema Registry)

Обоснование:

- Управление версиями схем событий
- Предотвращение несовместимых изменений
- Валидация событий перед публикацией
- Мониторинг использования версий схем

Использование:

- Регистрация и версионирование схем событий
- Валидация совместимости схем
- Метрики использования версий
