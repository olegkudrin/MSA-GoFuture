# Нефункциональные требования

## Производительность

### Пропускная способность

- Целевая нагрузка: обработка 500 тысяч конкурентных поездок
- Текущая нагрузка: 100 тысяч (каскадные отказы)
- Время отклика API: 95% < 200 мс для критичных операция (бронирование, поиск водителей)
- Время отклика API: 95% < 500 мс для некритичных операций (аналитика, уведомления)

### Масштабируемость

- Горизонтальное масштабирование: автоматическое добавление инстансов при нагрузке > 70% CPU
- Географическое распределение: поддержка нескольких регионов (несколько континентов)
- Автоматическое масштабирование: целевая утилизация ресурсов 60-80%

## Надежность и доступность

### Доступность

- Критичные сервисы: 99,95% uptime (Booking, Driver, Geography, Payments, Pricing)
- Некритичные сервисы: 99,9% uptime (Analytics, Notification, Fraud, Payouts)
- Среднее время восстановления (MTTR): менее 15 минут, для критичных сервисов менее 5 минут

### Отказоустойчивость

- Circuit breaker: автоматическое отключение деградировавших зависимостей
- Graceful degradation: работа при отказе некритичных компонентов
- Retry policies: экспоненциальная задержка с джиттером для внешних API
- Timeouts: жесткие тайм-ауты для всех внешних вызовов

### Резервирование

- Базы данных: развертываниe в нескольких зонах доступности (AZ) и автоматический failover
- Сервисы: миниум два инстанса для критичных сервисов

## Масштабируемость данных

### Объем данных

- Текущий объем: большой объем исторических данных в монолитной БД
- Рост: примерно 100 гигабайт новых транзакционных данных в месяц
- Аналитика: отдельное хранилище (ClickHouse) для историчеких данных

### Производительность БД

- Время отклика на запросы: 95% < 50 мс для OLTP
- Партиционирование: по географическим регионам и времени
- Индексация: оптимизированные индексы для частых запросов

## Безопасность

### Аутентификация и авторизация

- Identity Management: централизованная система управления идентификацией
- OAuth 2.0: для внешних партнеров и API
- JWT: для межсервисной коммуникации
- RBAC: ролевая модель доступа

### Защита данных

- Шифрование при хранении: шифрование персональных данных в БД
- Шифрование при передаче: использовать TLS для всех соединений
- Защита персональных данных: соблюдение всех регламентов и требований
- Управление секретами: централизованной хранилище секретов

### Аудит

- Логирование: все операции с финансовыми данными
- Трейсинг: распределенная трассировка запросов

## Наблюдаемость

### Мониторинг

- Метрики: полное покрытие критичных метрик (latency, error rate, throughput)
- SLI/SLO: определение показателей и мониторинг уровней обслуживания
- Алерты: многоуровневая система алертов (warning -> critical -> incident)

### Логирование

- Централизованные логи: агрегация логов всех сервисов
- Структурированное логирование: JSON-формат для машинной обработки
- Сроки хранения логов: хранение логов 30 дней (hot) и 180 дней (cold)

### Трейсинг

- Распределенная трассировка: сквозная трассировка запросов через микросервисы
- Частичная трассировка (sampling): для ошибок 100%, для успешных запросов 1-5%

## Развертывание и доставка

### CI/CD

- Время сборки: менее 10 минут для одного сервиса
- Время развертывания: менее 15 минут для rolling deployment
- Откат изменений: автоматический откат при ошибках менее чем за 5 минут
- Стратегия развертывания: сине-зеленое или канареечное для критичных сервисов

### Вермя выхода на рынок

- Целевое время: выход новых функций за две недели
- Независимое развертывание: каждая команда развертывает свой сервис независимо
- Переключатели функций: управление функционалом без повторного развертывания

## Интеграции

### Внешние системы

- Платежные шлюзы: Яндекс Пэй, локальные платежные системы
- Картографические сервисы: Яндекс Карты, локальные картографические API
- Push-уведомления: FCM, APNs, Huawei Push Kit
- Банковские API: интеграция для выплат водителям

### Требования к интеграциям

- Тайм-ауты: менее 5 секунд для синхронных вызовов
- Повторные попытки: автоматические повторы с экспоненциальной задержкой
- Резервная стратегия: резервные механизмы при недоступности внешних систем

## Совместимость и регуляторные требования

### Локальные требования

- Соблюдение нормативов: соответствие локальным законам о защите данных
- Локализация данных: хранение данных в соответствующих георграфических регионах
- Налоговая отчетность: интеграция с локальными налоговыми системами

### Обратная совместимость

- Версионирование API: поддержка нескольких версий API одновременно
- Поддержка устаревших API: минимум 6 месяцев поддержки устаревших версий
- Миграция между версиями API: постепенный переход без простоев

## Операционная эффективность

### Стоимость

- FinOps: мониторинг и оптимизация затрат на инфраструктуру
- Целевая экономия: снижение затрат на 40% за счет оптимизации ресурсов
- Распределение затрат: прозрачное распределение затрат по командам

### Опыт разработчиков (Developer Experience)

- Самообслуживание (Self-service): платформа для самостоятельного создания и развертывания сервисов
- Документация (Documentation): актуальная документация по API и архитектуре
- Локальная разработка (Local development): возможность локальной разработки и тестирования

## Специфичные бизнес-требования

### Динамическое ценообразование

- Цены в реальном времени: расчет цены в режиме реального времени с учетом спроса и предложения
- Временные параметры: расчет цены менее чем за 100 мс

### Оптимизация загрузки водителей

- Умное распределение: алгоритм предотвращения "горячих зон"
- Модели машинного обучения: прогнозирование спроса и оптимизация назначения водителей

### Мультиарендность

- Партнерская программа: поддержка внешних партнеров через API
- Изоляция данных: логическое разделение данных партнеров
- Брендирование под партнера: возможность кастомизации для партнеров
