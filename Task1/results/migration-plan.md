# План миграции от монолита к микросервисам

## Введение

Что мигрируем:

- Монолитное приложение Django -> девять независимых микросервисов
- Единая база данных PostgreSQL -> отдельные БД для каждого сервиса
- Переносим порядка одного терабайта данных
- RabbitMQ + Celery -> Event-driven архитектуру на Kafka

Целевое состояние:

- Микросервисная архитектура с четкими доменными границами
- Свой набор баз данных у каждого микросервиса
- Event-driven коммуникация через Kafka
- Kubernetes для оркестрации
- Горизонтальное автомасштабирование

## Этапы миграции по кварталам

### 1 квартал 2026: Фундамент (8 недель)

Сервисы: Geography Service, Notification Service

Что делаем:

- Развертывание инфраструктуры (K8s, API Gateway, Kafka, Observability)
- Разработка сервисов (геопоиск, уведомления)
- Миграция данных (~7GB): двойная запись, синхронизация, переключение трафика
- Постепенное переключение: 10%, 50%, 100%

Критерии успеха:

- Время геопоиска < 50 мс (95%)
- Независимые развертывания сервисов
- Event-driven архитектура работает

### 2 квартал 2026: Критичные бизнес-домены (14 недель)

Сервисы: Pricing Service, Driver Service, Booking Service

Что делаем:

- Разработка критичных доменов (ценообразование, водители, бронирования)
- Миграция данных (~250GB): CDC (Debezium) для репликации в реальном времени
- Интеграция серисов через REST API и Kafka
- Канареечное развертывание Booking: 1%, 5%, 25%, 100%

Критерии успеха:

- 300 тыс. конкурентных поездок через новые микросервисы
- Время расчета цены < 100 мс
- Реализация Saga для бизнес-процессов

### 3 квартал 2026: Финансовые сервисы (11 недель)

Сервисы: Payments Service, Payouts Service

Что делаем:

- Разработка финансовых сервисов с учетом всех требований
- Миграция данных (~110GB): двойная запись + сверка данных
- Интеграция с Яндекс Пэй и банковскими API
- Двойная обработка платежей с ежедневной сверкой

Критерии успеха:

- Нулевая потеря транзакций (100% согласованность)
- Соответствие регуляторным требованиям
- Аудит всех операций

### 4 квартал 2026: Аналитика и безопасность (10 недель)

Сервисы: Fraud Service, Analytics Service

Что делаем:

- Разработка ML-сервисов (определение мошенничества)
- Миграция аналитики в ClickHouse (~510GB): выгрузка + потоковая передача
- Настройка Kafka Connect для CDC из всех сервисов
- Дашбоарды в реальном времени в DataLens (BI-система)

Критерии успеха:

- Определение мошенничества в реальном времени (< 500 мс)
- Аналитика в реальном времени (задержка < 5 сек.)
- 500 тыс. поездок, 99,95% uptime

## Стратегия миграции

Инфраструктура:

- Kubernetes в облаке (Yandex Cloud) для оркестрации
- API Gateway (Strangler Fig) для постепенного вытеснения монолита
- Kafka как Event Bus для асинхронной коммуникации
- Observability: Prometheus, Grafana, Jaeger, Loki

Разработка сервисов:

- Database per Service - изоляция данных
- Saga Pattern - распределенные транзакции
- Circuit Breaker - устойчивость к сбоям в зависимостях
- Feature Toggles - постепенное включение функциональности

Миграция данных:

- Двойная запись: монолит пишет в обе БД (старая + новая)
- CDC (Change Data Capture): Debezium для репликации в реальном времени
- Пакетная миграция: Spark jobs для исторических данных
- Сверка данных: ежедневно сверяем данные, автоматическое исправление расхождений

Переключение трафика:

- Канареечное развертывание: Shadow mode, 1%, 10%, 50%, 100%
- Мониторинг: метрики по latency, error rate, расхождения данных
- Критерии переключения: расхождение < 0,01%, latency не ухудшилась, error rate < 0,1%

## Риски и откат

Основные риски:

- Несогласованность данных между монолитом и сервисами
- Увеличение latency из-за межсервисной коммуникации
- Потеря транзакций (критично для финансовых сервисов)
- Каскадные отказы при сбое критичного сервиса

Стратегия отката:

| Фаза миграции      | Откат                                          | RTO      | RPO     |
| ------------------ | ---------------------------------------------- | -------- | ------- |
| Синхронизация      | Остановка репликации, монолит работает         | < 5 мин  | 0       |
| Canary (< 10%)     | Переключение трафика на момнолит через Gateway | < 10 мин | 0       |
| Частичная (10-50%) | Обратная двойная запись + переключение         | < 20 мин | < 5 мин |
| Полная (> 50%)     | Обратная двойная запись + постепенный откат    | < 30 мин | < 5 мин |

Критерии отката:

- Error rate более 1% в течение 10 минут
- Latency увеличились в 2 раза в 95%
- Расхождение данных более 0,5%
- Потеря транзакций (для финансовых сервисов)

## Критерии завершения миграции

Технические:

- Все девять сервисов развернуты и стабильны
- Монолит обрабатывает 0% продуктового трафика
- Данные мигрированы с расхождением менее 0,001%
- Нагрузочный тест пройден (500 тыс. поездок, менее 200 мс в 95%)

Бизнес-метрики:

- 99,95% uptime критичных сервисов
- MTTR менее 15 минут
- Время выхода новых функций менее двух недель
- Горизонтальное масштабирование в трех или более регионах

## Заключение

План миграции рассчитан на 1 год с поквартальым выделением сервисов. Используется паттерн Strangler Fig для постепенного вытеснения монолита без остановки системы. Критичные финансовые сервисы мигрируются в третьем квартале с максимальными гарантиями согласованности данных. К концу года платформа GoFuture полностью переходит на микросервисную архитектуру с возможностью обработки 500 тыс. конкурентных поездок и горизонтальным масштабированием в трех или более географических регионах.
