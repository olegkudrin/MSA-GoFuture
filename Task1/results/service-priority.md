# Очередность выделения сервисов

## Набор сервисов

Выделяем сервисы для каждого компонента монолитного приложения (Django Application).

| Название     | Описание                  |
| ------------ | ------------------------- |
| Booking      | Управление бронированиями |
| Driver       | Управление водителями     |
| Pricing      | Ценообразование           |
| Payments     | Платежи от пассажиров     |
| Payouts      | Выплаты водителям         |
| Notification | Управление уведомлениями  |
| Geography    | Геопоиск и маршрутизация  |
| Analytics    | Сбор аналитики            |
| Fraud        | Обнаружение мошенничества |

## Принципы приоритизации

При определении очередности выделения сервисов учитываются следующие критерии:

1. Бизнес-критичность - влияние на основной денежный поток и выручку
2. Сложность интеграции - количество зависимостей с другими доменами
3. Частота изменений - потребность в быстром изменении функциональности
4. Риски миграции - объем данных и сложность переноса
5. Готовность команды - наличие выделенной продуктовой команды

## Фазы миграции

### Фундамент

#### Geography Service (высокий приоритет)

Обоснование:

- Низкая связность с остальными доменами (используется другими, но сам независим)
- Четкая доменная граница - работа с геоданными, маршрутизацией, поиском водителей
- Интеграция только с внешними API (Яндекс Карты) и Elasticsearch
- Минимальные риски - небольшой объем персистентных данных
- Предоставит опыт команде перед миграцией более сложных сервисов

Зависимости: Яндекс Карты API, Elasticsearch

Ожидаемый эффект: Улучшение производительности геопоиска, независимое масштабирование

#### Notification Service (высокий приоритет)

Обоснование:

- Полностью асинхронная логика через очереди (уже использует Celery)
- Не блокирует критичные бизнес-процессы - может деградировать без остановки системы
- Простая миграция данных - минимальный объем (настройки устройств, шаблоны)
- Интеграция с внешними push-сервисами (FCM, APNs, Huawei)
- Позволит снизить нагрузку на монолитное приложение

Зависимости: FCM, APNs, Huawei Push Kit, Event Bus

Ожидаемый эффект: Изоляция сбоев уведомлений, улучшение процента доставки

### Критичные бизнес-домены

#### Pricing Service (критичный приоритет)

Обоснование:

- Критичный для динамического ценообразования в реальном времени
- Высокая частота изменений - постоянные эксперименты с ценообразованием
- Небольшой объем данных (правила ценообразования, коэффициенты)
- Используется всеми: Booking, Driver, Payments
- Требует независимого масштабирования для обработки пиковых нагрузок

Зависимости: Event Bus

Ожидаемый эффект: Адаптация цен в реальном времени, быстрое внедрение наработок по ценообразованию

#### Driver Service (критичный приоритет)

Обоснование:

- Ядро бизнес-логики - управление водителями, их статусами, доступностью
- Средний объем данных, но критичных для бизнеса
- Используется: Booking (поиск и назначение), Payouts (данные для выплат)
- Высокая частота изменений - постоянное развитие возможностей для водителей
- Требует независимого масштабирования - обеспечение быстрого поиска водителей

Зависимости: Geography Service, Event Bus

Ожидаемый эффект: Независимое развитие функциональности для водителей, улучшение процесса подключения водителей к системе.

#### Booking Service (критичный приоритет)

Обоснование:

- Самый критичный домен - оркестрация всего процесса поездки
- Высокая сложность - интеграция со всеми остальными сервисами
- Большой объем транзакционных данных (история поездок)
- Наивысший приоритет для бизнеса (генерация выручки)
- Требует всех предыдущих сервисов для успешной миграции

Зависимости: Driver, Pricing, Geography, Fraud, Notification (через Event Bus)

### Финансовые и платежные сервисы

#### Payments Service (высокий приоритет)

Обоснование:

- Финансово-критичный домен - платежи от пассажиров
- Требования законодательства - необходима изоляция финансовых данных
- Интеграция с внешними платежными шлюзами (Яндекс Пэй, локальные системы)
- Средний объем данных, высокие требования к безопасности и аудиту
- Связан с системой выявления мошенничества (Fraud)

Зависимости: Яндекс Пэй, Fraud Service, Event Bus

Ожидаемый эффект: Улучшение безопасности, быстрая интеграция новых платежных методов

#### Payouts Service (высокий приоритет)

Обоснование:

- Финансово-критичный домен - выплаты водителям
- Асинхронная обработка через Celery (упрощает миграцию)
- Законодательство и аудит - требуется отдельная изоляция
- Интеграция с банковскими API
- Связан с Driver Service для получения данных

Зависимости: Driver Service, Banking API, Event Bus

Ожидаемый эффект: Прозрачность выплат, соответствие локальным регуляторным требованиям

### Аналитика и безопасность

#### Fraud Service (средний приоритет)

Обоснование:

- Неблокирующий сервис - работает асинхронно с основным потоком обработки
- Модели машинного обучения требуют изолированного окружения и вычислительных ресурсов
- Средний объем данных (правила, модели, история инцидентов)
- Интегрируется с Payments и Booking для проверок
- Можно выделить позже, когда основные домены стабилизированы

Зависимости: Event Bus (события от Booking, Payments), платформа машинного обучения

Ожидаемый эффект: Улучшение механизмов определения мошенничества, снижение финансовых потерь

#### Analytics Service (низкий приоритет)

Обоснование:

- Полностью некритичный для операционных процессов
- Асинхронная обработка данных через Spark / Flink
- Большой объем исторических данных (требует аккуратной миграции в ClickHouse)
- Может работать поверх Event Bus, собирая события от всех сервисов
- Выделяется последним, когда event-driven архитектура стабильна

Зависимости: Event Bus (все события), ClickHouse, DataLens

Ожидаемый эффект: Аналитика в реальном времени, независимое развитие BI-инструментов

## Механизм обратной совместимости

Для всех сервисов применяется паттерн Strangler Fig:

- API Gateway - единая точка входа, маршрутизация запросов между монолитом и новыми сервисами
- Feature toggles - гибкое управление включением/выключением новой функциональности
- Database replication - синхронизация данных между монолитом и новыми БД во время миграции
- Event Bridge - монолит публикует события для новых сервисов, новые сервисы публикуют события для монолита
- Versioned API - поддержка старых версий API минимум 6 месяцев

## Критерии успеха по фазам

- География и уведомления работают независимо, время геопоиска менее 50 мс.
- Процесс бронирования работает через микросервисы, обработка 300 тыс. поездок.
- Платежи изолированы, соответствие локальным регуляторным требованиям.
- Полная микросервисная архитектура, 500 тыс. поездок, 99.95% uptime.
