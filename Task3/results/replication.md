# Схема репликации данных между регионами

## Базы данных доменных сервисов

- Каждый доменный сервис (их девять) имеет свою базу данных в каждом активном регионе (database-per-service-per-region).

- Для операций в онлайне используется локальная запись в региональный кластер БД (минимальная задержка, высокая доступность внутри региона: multi-AZ + автоматический failover).

- Между регионами настраивается асинхронная кросс-регионная репликация:

  - для критичных доменов с транзакционными и персональными данными (Payments, Payouts, Booking, Driver, Pricing) - двусторонняя репликация для взаимного резервирования между EU/SEA/SA и односторонняя репликация для аварийного восстановления со всех регионов в DR.

  - для бизнес-критичных конфигурационных доменов (Fraud) - репликация в DR (EU/SEA/SA -> DR): правила фрода важны для финансовой безопасности, потеря данных приведет к финансовым потерям, асинхронная работа позволяет обойтись репликацией только в DR.

  - для некритичных конфигурационных доменов (Geography, Notification) - межрегионная репликация не требуется: данные можно восстановить из внешних API (Яндекс Карты) или пересоздать из конфигурационных хранилищ за 30–120 минут; Geography критичен (99.95% uptime), но данные восстанавливаются быстро через API + кэш; при необходимости возможна опциональная репликация настроек в DR.

  - для аналитики - односторонний экспорт из всех регионов в глобальный аналитический контур (EU/DR), с агрегацией и анонимизацией.

- Технологически: PostgreSQL logical replication / CDC (Debezium) + потоковая передача в целевые кластеры.

## Kafka и события

- В каждом регионе - свой Kafka-кластер, все доменные сервисы пишут/читают локальные топики \*.{region}.

- Для глобального бэкапа и межрегионной аналитики используется MirrorMaker 2:
  - реплицирует ключевые топики booking, payments, payouts, driver, pricing, fraud из регионов SEA/SA в EU/DR;
  - репликация асинхронная, не участвует в синхронном критичном пути пользователя.

## Стратегия согласованности и RPO/RTO

- Операции внутри региона - сильная согласованность (master + sync-реплики в рамках одного региона/нескольких AZ).

- Между регионами - eventual consistency, RPO по критичным доменам менее 1–5 минут (зависит от задержки CDC / MirrorMaker), RTO при переключении региона 15–30 минут.

- Внутри бизнес-процессов используется Saga, межрегионная репликация не участвует в транзакционном контуре.
