# Механизмы обеспечения качества данных

## Обзор

Для обеспечения качества данных в пайплайне ML-платформы и BI-инструментов используются многоуровневая система валидации и контроля качества данных на всех этапах обработки.

## 1. Валидация схем событий (Schema Registry)

### Механизм

- Confluent Schema Registry обеспечивает централизованное управление схемами событий
- Все микросервисы регистрируют схемы событий перед публикацией
- Stream Processor валидирует схемы при чтении событий из Kafka

### Преимущества

- Гарантия совместимости схем между версиями
- Автоматическое отклонение событий с невалидными схемами
- Версионирование схем для обратной совместимости

## 2. Проверка качества данных в реальном времени (Data Quality Service)

### Механизм

Great Expectations используется для проверки качества данных на этапах:

1. Входные данные (Kafka)

   - Проверка обязательных полей
   - Валидация типов данных
   - Проверка диапазонов значений
   - Обнаружение дубликатов

2. Обработанные данные (Stream Processor)

   - Проверка полноты агрегаций
   - Валидация бизнес-правил
   - Проверка временных меток

3. Данные для ML (Processed Data Store)

   - Проверка распределений признаков
   - Обнаружение аномалий в данных
   - Проверка отсутствия пропусков в критичных фичах

### Примеры проверок

- BookingCreated: обязательные поля (bookingId, userId, timestamp), валидация координат
- DriverLocationUpdated: проверка координат в допустимом диапазоне, временная метка не в будущем
- PriceCalculated: цена > 0, валидация валюты

## 3. Мониторинг качества данных

### Метрики

- Completeness: процент заполненности обязательных полей
- Validity: процент событий, прошедших валидацию схем
- Timeliness: задержка между созданием события и его обработкой
- Consistency: согласованность данных между сервисами
- Accuracy: точность данных (сравнение с эталонными источниками)

### Инструменты мониторинга

- Prometheus: сбор метрик качества данных
- Grafana: визуализация метрик и дашборды
- Alertmanager: алерты при деградации качества данных

## 4. Обработка ошибок и Dead Letter Queue (DLQ)

### Механизм

- События, не прошедшие валидацию, отправляются в DLQ
- Автоматические retry с экспоненциальной задержкой
- После исчерпания попыток - отправка в DLQ для ручного анализа
- Уведомления командам разработки о проблемах с данными

### Процесс восстановления

1. Анализ событий в DLQ
2. Исправление схем или данных в источнике
3. Репроцессинг событий из DLQ после исправления

## 5. Качество данных для ML-моделей

### Feature Store

- Валидация фич: проверка распределений, отсутствие выбросов
- Версионирование фич: отслеживание изменений в фичах
- Мониторинг дрифта: обнаружение изменений в распределении данных

### Model Registry

- Версионирование моделей: отслеживание версий моделей и данных обучения
- A/B тестирование: сравнение производительности моделей
- Мониторинг деградации: отслеживание метрик моделей в production

## 6. Качество данных для BI

### ClickHouse

- Проверка агрегаций: валидация сумм, счетчиков, средних значений
- Временная согласованность: проверка временных меток и интервалов
- Данные для отчетов: валидация перед загрузкой в DataLens

### Analytics Service

- ETL валидация: проверка данных на этапе трансформации
- Регрессионное тестирование: сравнение результатов с предыдущими периодами
- Аномалии в данных: обнаружение неожиданных изменений в метриках

## 7. Автоматизация проверок качества

### CI/CD интеграция

- Автоматическая проверка схем при деплое сервисов
- Валидация тестовых данных в staging окружении
- Проверка качества данных перед релизом моделей

### Планирование проверок

- Ежедневные проверки: полная валидация данных за прошедший день
- В реальном времени: проверка критичных событий при поступлении
- Еженедельные отчеты: анализ качества данных и трендов

## 8. Метрики успеха

- Процент валидных событий: > 99,9%
- Время обнаружения проблем: < 5 минут
- Время восстановления: < 30 минут
- Процент данных для ML: > 99% проходят валидацию
- Точность BI-отчетов: 100% валидация перед публикацией
